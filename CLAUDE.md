# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Purpose

This is a **learning exercise**, not a production codebase. The deliverable is a working prototype of a dynamic QR code / URL shortener service that satisfies the spec in `PROMPT.md`. The user has chosen the **Challenge Track**: implementation is built from scratch — file layout, framework choice, and architecture are all open.

`PROMPT.md` is the source of truth for requirements. `README.md` describes how the exercise is organized. `scaffold/` contains the Guided Track boilerplate (FastAPI + SQLAlchemy + SQLite) and is **reference material only** under the Challenge Track — do not edit files in `scaffold/` unless the user explicitly asks.

## Working on This Exercise

When the user asks for implementation help, treat `PROMPT.md` as the spec:

- **System Requirements** (PROMPT.md lines 5-12): create / redirect (302) / get-info / update / soft-delete / optional expiration / URL validation / 410 vs 404 distinction.
- **Design Questions** (PROMPT.md lines 16-26): the user is expected to answer these themselves before/while coding — these are discussion material for the live session, not for Claude to answer unsolicited.
- **Verification curl tests** (PROMPT.md lines 32-77): the prototype is "done" when all of these pass. Use them as the acceptance test.

Required endpoints (exact paths matter — the curl tests hard-code them):

- `POST /api/qr/create` → returns `{token, short_url, qr_code_url, original_url}`
- `GET /r/{token}` → 302 redirect, 410 if deleted/expired, 404 if not found
- `GET /api/qr/{token}` → metadata
- `PATCH /api/qr/{token}` → update target URL and/or `expires_at`
- `DELETE /api/qr/{token}` → soft delete (200)
- `GET /api/qr/{token}/image` → PNG of the QR code
- `GET /api/qr/{token}/analytics` → `{token, total_scans, scans_by_day}`

The 410-vs-404 split is a real requirement: deleted/expired tokens return **410 Gone**; tokens that never existed return **404 Not Found**. Don't collapse them.

## Reference: What scaffold/ Already Decided

If the user wants to reuse scaffold decisions on the challenge side, these are the choices baked into `scaffold/`:

- Python 3.10+, FastAPI, SQLAlchemy 2.x (Mapped/mapped_column style), SQLite (`./qr_code.db`), `qrcode[pil]` for image generation.
- Two tables: `url_mappings` (token PK lookup, `is_deleted`, `expires_at`) and `scan_events` (analytics, indexed on `(token, scanned_at)`).
- Token: 7-char Base62, generated by SHA-256(url + nonce) → base62 → truncate, with collision retry against the DB.
- Redirect path uses an in-memory cache (simulating Redis) with cache invalidation on update/delete.
- `BASE_URL = "http://localhost:8000"` is hardcoded — the curl tests assume port 8000.

These are conventions, not requirements. The Challenge Track is free to diverge (different language, different storage, different token scheme) as long as the curl tests pass.

## Running the Prototype

If implementing in the recommended stack:

```bash
# from the project root, in the directory containing your app
python3 -m venv .venv
source .venv/bin/activate
pip install fastapi uvicorn sqlalchemy 'qrcode[pil]'
uvicorn app.main:app --reload   # adjust module path to your layout
```

Verification (after the server is up on port 8000): run the curl block at the bottom of `PROMPT.md` end-to-end. A passing run hits 200/302/410/404 in the right places.
